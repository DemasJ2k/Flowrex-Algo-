### 5.9 ⭐ Dukascopy-direct backtest (NEW — user request)
**User requirement:** "For backtest, if there is a possible way for us to always be drawing our backtest data from Dukascopy. I do not want the file they fetch to stay on the database for a long period of time."

**Design:**
- New class `BacktestDataFetcher` at `backend/app/services/backtest/data_fetcher.py`
- On backtest start, spawns the Dukascopy Node fetcher to write a **tempfile** at `/tmp/flowrex-backtest/{backtest_id}/{symbol}_{tf}.csv`
- Backtest engine reads from those tempfiles (same existing CSV interface)
- On backtest completion (success OR failure), cleanup hook deletes the tempdir
- Short-lived in-memory cache (60 min TTL) so concurrent backtests for the same symbol+tf+date-range reuse the same fetch
- Fetch tempfiles NEVER touch the postgres DB
- A janitor task on startup removes any orphaned `/tmp/flowrex-backtest/*` directories older than 24h (handles crashes)

**Files to modify:**
- `backend/app/services/backtest/data_fetcher.py` — new
- `backend/app/api/backtest.py` — replace `m5_path = os.path.join(DATA_DIR, f"{body.symbol}_M5.csv")` with `data = fetcher.fetch(body.symbol, timeframes=["M5","H1","H4","D1"], days=2500)`
- `backend/app/services/backtest/engine.py` — accept a data bundle instead of file paths
- Cleanup hook wired to FastAPI `BackgroundTasks`

**Rationale for tempfile over pure in-memory:**
- The existing backtest engine reads CSVs with pandas; tempfile preserves that interface with minimal refactor
- If backtests become large, tempfile avoids holding 1M+ rows in memory across concurrent users
- Cleanup is trivial (`shutil.rmtree(tempdir)`)

**Note:** This does NOT replace training. Training still uses the persistent `/opt/flowrex/History Data/data/` files. Only the backtest UI flow changes.
